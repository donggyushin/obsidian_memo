# Harness Engineering

`claude code`, `codex` 에 이미 내장되어있는 하네스가 있음. 내 프로젝트에 맞는 커스텀 하네스를 한 층 더 쌓아올리면 성능이 더 좋아짐.

## 커스텀 하네스

`CLAUDE.md`, `docs/`, `hooks/` — 내 프로젝트의 규칙, 아키텍쳐, 기술 선택을 한 층 더 올리는 것.

### oh-my-claudecode

이런 오픈소스를 가져다 쓰면 바로 2층이 올라감. 시작하기 좋은 범용 하네스. AI 가 일하는 방식 자체를 되게 체계적으로 만들어줌.

![[스크린샷 2026-04-20 오후 5.16.41.png]]

### 커스텀 하네스 핵심 요소

- `CLAUDE.md` 에 프로젝트 규칙
- `docs/` 에 기획서 + 아키텍쳐
- `hooks` 으로 자동 검증
- 가볍고 바로 시작 가능

## 기획단계의 중요성

기획단계가 엄청 중요하다고 함.

OpenAI 직원 3명이 5개월동안 코드 0줄 쓰고 제품을 출시함. AI 가 안전하게 코드를 작성할 수 있게 규제를 강하게 걸어놓고 AI 를 활용함.

![[스크린샷 2026-04-20 오후 6.45.46.png]]

## 왜 기존 방법이 실패할까?

AI를 잘 활용하는 방법 — AI 가 일할때 필요한 내용만 정확하게 주기.

근데 AI 가 정보는 전부 알고 있는데, 자기 마음대로 행동하는 경우가 있음. **규칙과 울타리의 문제**임. AI는 통제없이는 위험.

- **에이전틱 엔지니어링**: 에이전트를 더 똑똑하고 강하게 만드는 기술
- **하네스 엔지니어링**: 에이전트를 더 통제하고 규율을 거는 기술

![[스크린샷 2026-04-20 오후 6.49.02.png]]

## 하네스 엔지니어링의 본질

AI 에이전트가 자율적으로 일하되, 동시에 안전하게 통제할 수 있는 환경을 만드는 것.

전통적으로는 같은 인풋 같은 아웃풋으로 예측이 가능했는데, AI 시대에는 같은 인풋이 들어와도 아웃풋이 항상 다르게 나오기 때문에 규칙을 강하게 걸어두는 것임. **실패가 구조적으로 반복 불가능하도록** harness 를 거는 것. 시스템적으로 설계를 하는 것임.

## 4개의 기둥

### 1. 기계가 읽는 컨텍스트

코드 저장소 안에 AI가 읽는 런타임 설정 파일을 배치.

### 2. 자동 교정 루프 (CI/CD 게이트)

AI 는 규칙을 읽고도 무시할 수 있음. 린터(Linter), 구조 테스트, Pre-commit hook 과 같이 자동 교정 루프를 만들어두기. 시스템 자체가 실수하지 못하도록 막음.

### 3. 명시적 도구 경계

- **파일시스템**: `src/` 읽기·쓰기 OK, `config/` 읽기만
- **API**: 내부 API OK, 외부 서비스 불가
- **데이터베이스**: `SELECT` OK, `DROP TABLE` 불가
- **시스템**: 화이트리스트 명령만 허용

### 4. 지속적 피드백 루프 (Garbage Collection)

진화 사이클. 시간이 지날수록 시스템이 더더욱 견고해짐.

1. 규칙 위반 감지
2. 자동 리팩토링
3. 코드 정리
4. 아키텍처 체크

> [!tip] 핵심 원칙
> 부탁은 안할 수 있지만, 규칙은 지켜야함.

프로젝트 전체를 AI 안전 환경으로 만들기. AI 가 실수할 수 없는 환경으로 만드는 것. 

규칙과 제한을 둬서 애초에 잘못된 방향으로 가지 않게.
만약 가더라도 같은 실수가 반복되지 않게.
우리의 간섭이 필요없는 시스템을 만드는 것.

AI 에이전트가 실수했을 때, 프롬프트를 고치지 마세요. 마구를 고치세요. 그것이 하네스 엔지니어링의 핵심.
