## 핵심 요약

Claude Code 를 매일 800시간 가까이 쓴 인디 개발자의 회고. 구조는 **3×3** — *하지 말 것 3 / 해야 할 것 3 / 실전 팁 3*. 일관 메시지는 **"신선한 컨텍스트가 비대한 컨텍스트를 이긴다"** 와 **"AI 가 알아서 잘하길 바라지 말고 명시적으로 시켜라"**. 보리스(Claude Code 창시자)의 vanilla 세팅 발언, "단 하나의 팁을 고르라면 검증" 발언, Anthropic 의 "AI 자율 ≠ 인간 검증 패키지" 입장이 인용 출처로 등장.

## Part 1 — 하지 말 것 3가지

### 1) 처음부터 풀 세팅 금지

이런저런 스킬 · 라이브러리 · 플러그인을 처음부터 다 깔지 말 것. **Claude Code 창시자 보리스** 본인이 X 에서 *"내 세팅은 놀랍게도 vanilla 다"* 라고 공개. 본인이 만든 툴인데도 거의 커스텀 안 함.

**권장 순서**:

1. 기본 Claude Code 부터 사용
2. 같은 실수 / 같은 워크플로우를 반복하는 패턴이 보이면
3. **그 패턴 하나에 대해서만** 스킬 작성

→ 실제 니즈에 따라 자라나는 세팅이 가장 좋은 세팅. *"좋은 거 다 갖다 박는 게 결코 좋은 게 아닙니다."*

### 2) FOMO 금지

Claude 업데이트 속도가 매우 빠르다 (루틴, 데스크 맵 리디자인, 클로드 디자인 등 매주 신기능 쏟아짐). 세 기능을 무시하라는 게 아니라, **"다 따라가야 한다는 불안에 휩쓸리지 말 것"**. 당장 워크플로우와 안 맞거나 필요 없을 확률이 높다.

태도의 문제로 다시 다룸 → Part 3 의 마지막 팁 (열린 사고 vs FOMO).

### 3) "알아서 해줘" 프롬프트 금지

*"이거 만들어 줘"*, *"이 버그 알아서 고쳐 줘"* 류. AI 는 유능하지만 **단번에 완성도 있는 결과물이 나오는 경우는 거의 없다**.

Anthropic 공식 입장 인용:

> AI 가 혼자 알아서 다 하는 방향이 아니라, **인간이 이미 검증한 방식을 패키지로 만들어서 AI 한테 시키는 방향**이 맞다.

**자율주행 비유** — 자율주행 차를 타도 *어디로 갈지* 는 사람이 정한다. 내비게이션 정도는 직접 찍어야 한다.

## Part 2 — 해야 할 것 3가지

### 1) 컨텍스트 관리

> **신선한 컨텍스트가 비대한 컨텍스트를 이긴다.**

이 한 문장만 기억해도 됨. Claude 의 컨텍스트는 메시지·파일·실행 명령어가 모두 쌓이며, **채워질수록 성능 저하**.

- 정보를 **많이 주는 게 아니라** 정말 필요한 것만 남기고 나머지를 쳐낸다
- 관련 없는 정보가 많을수록 Claude 가 잘못된 방향으로 빠짐
- **CLAUDE.md 도 80% 정도만 따른다**고 알려져 있음 — 길수록 중요 규칙이 무시됨
- 권장 길이: **150~200줄 안에서 끝낼 것**

→ [[concepts/fresh-context-principle]], [[concepts/context-engineering]], [[concepts/claude-md]].

### 2) 진짜 계획

여기서 말하는 계획은 **플랜 모드 켜라는 게 아니라** 진짜 계획. 어떤 AI 든 *"여러분이 뭘 만들고 싶은지 명확히 알면"* 그 AI 가 뭐라도 한다.

- *"이 기능 추가해 줘"* (X) → **어떤 식으로 구현할지, 완료 기준이 뭔지 먼저 정리** (O)
- 플랜 모드의 가치는 *이 과정을 잘 몰라도 Claude 와 함께 할 수 있다*는 점
- 단 많은 사용자가 플랜을 그냥 수락 → **편집을 적극적으로 할 것**
- **명시적으로 지시**하라: 스킬을 만들어 놨어도 *"이거 구현해 줘"* 만 던지면 Claude 가 그 스킬을 안 쓰고 자기 멋대로 함

> AI 가 가진 도구·스킬·규칙을 진짜 쓰게 하려면, **알아서 잘해 주길 바라지 말고 대놓고 시켜라.**

→ [[concepts/plan-mode]].

### 3) 검증

**보리스가 단 하나의 팁을 고르라 했을 때 고른 것**. Claude 한테 *자기 일을 스스로 검증할 수 있는 방법* 을 주기만 해도 결과물 품질이 **2~3배** 올라감.

- 검증 장치 없으면 사용자가 유일한 피드백 루프 → 매 사이클을 직접 돌려야 함
- 검증 장치 있으면 Claude 가 스스로 돌리고 실패 시 스스로 고친 후, 사용자에겐 끝난 버전이 옴

> [!warning] Reward Hacking 주의
> 테스트가 실패하면 **코드를 고치는 게 맞지만**, AI 가 **테스트 자체를 코드에 맞춰 수정**하는 경우 존재. 통과는 됐지만 의미 없는 테스트.
>
> → 검증 루프를 만들어줬다고 끝이 아님. **테스트 자체가 올바른 걸 검증하는지** 사용자가 봐야 한다.

→ [[concepts/reward-hacking]], [[concepts/harness-engineering#3-execution-loop]].

## Part 3 — 실전 팁 3가지

### 1) 세션 관리 — 리와인드 우선

Claude Code 팀 인용: *"컨텍스트 관리 잘하는 사람을 가르는 단 하나의 습관을 꼽자면 그건 **리와인드**다."*

- **리와인드** = 대화 중 `Esc` 두 번 → 이전 메시지 지점으로 돌아가서 그 이후 기록을 싹 지움
- 시나리오: A 시도 실패 → *"B 로 해 줘"* 메시지 보내면 **A 실패 기록이 컨텍스트에 그대로 남아** Claude 가 영향받음
- 리와인드 후 B 시도 = A 흔적 없는 깨끗한 상태 → 결과물이 훨씬 깨끗
- `/clear` — 작업 끝났을 때 마침표 찍듯 세션 초기화
- `/compact` — 지금까지 대화 요약. **방향을 직접 지시할 것**: *"/compact 이번엔 A 에 집중하고 B 는 버려"*

→ [[concepts/rewind-pattern]].

### 2) Plugins / Skills 생태계

Claude 생태계가 이 방향으로 움직이는 중. 스킬은 특정 작업에 대한 **레시피** + **필요할 때만 로드**.

- CLAUDE.md = 세션 시작 시 **무조건** 컨텍스트에 들어감
- Skills = 실제 필요할 때만 로드
- **남이 만든 좋은 스킬을 가져다 쓰는 게 우선**. GitHub 에 유용한 스킬 多. 디스텍 스퍼파\* 같은 유명 리포지토리 존재
- 자기가 반복하는 작업이 있으면 그때 스킬로 저장
- 스킬은 **유지 비용이 거의 없으면서도 유용** → 어느 선에선 자산

\* 영상 발음 옮긴 표현. 실제 명칭 미확인 (`[!question]` 후속 lint 시 검증).

→ [[concepts/claude-skills]].

### 3) 태도 — 열린 사고 vs FOMO

기술적 팁이 아닌 태도. Part 1의 "FOMO 금지"와 모순처럼 들리지만 **다른 개념**:

| | FOMO | 열린 사고 |
|---|---|---|
| 본질 | 불안 | 사고 |
| 행동 | 안 쓸 거면서 일단 깔고 봄 | 왜 생겨났는지 · 내 프로젝트에 어떻게 적용할지 질문 |
| 시간이 갈수록 | 부담 누적 | 자산 누적 |

> 진짜 좋은 건 **남의 워크플로우를 그대로 복사하는 게 아니라**, 그걸 이해해서 **내 상황에 맞게 번역**해서 적용하는 능력. 자동화할 수 없는 부분이자 시간이 지날수록 자산이 되는 부분.
