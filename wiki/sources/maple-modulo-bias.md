## 핵심 요약

메이플스토리에서 발견된 **드랍률 표기 < 실제 드랍률** 버그의 기술적 원인 해설. 원인은 고전적 **[[concepts/modulo-bias]]** — C `rand()` 유의 PRNG 출력을 모듈로(`%`)로 잘라 작은 범위로 매핑할 때, `rand_max + 1` 이 divisor 로 나눠떨어지지 않으면 낮은 값 구간이 과대표집됨. 메이플은 `rand() % 10억` 경로를 사용했고 rand 범위가 약 42억이라 **0~2.9억 구간이 초과 1회씩 등장** → **29% 이하 드랍률 아이템의 실제 확률이 표기보다 높았음**. 동일 원인으로 드롭률 버프(+5/+10%)가 거의 무효했고, **17% 이상 올려야** 유의미한 변화가 발생. 유저에겐 개이득, 넥슨엔 손실.

## 주요 주장 · 데이터 · 인용

### 문제 요약

- 메이플 키우기 확률 문제 전수조사 → 오리지널 메이플스토리에서도 같은 계열 버그 확인.
- **표기된 드랍률보다 실제 드랍률이 훨씬 높았던 버그**. 쌀값(게임 내 재화 시세) 하락 말고는 유저 체감 손해 없음.
- 원인: **Modulo Bias** (모듈로 편향).

### 기술 구조 (영상 설명 그대로)

1. 예전 C 계열 구현: `rand()` 는 0 ~ 약 42억 9천만(`RAND_MAX` 상한, 플랫폼별로 상이) 범위 균등 난수 반환.
2. 확률 구현 시 모듈로로 범위 축소: `rand() % 100` → 0~99.
3. `result < 1` 이면 당첨 → 1% 확률. `result < 2` → 2%. 직관적·간편해서 옛 게임에서 관행.
4. 메이플은 `rand() % 10억` 으로 0~10억 구간을 먼저 뽑은 뒤 확률 계산.

### 왜 편향이 생기는가

rand 범위가 `[0, 8]` (9개), divisor 가 `6` 이라고 단순화하면:

| rand | `% 6` |
|------|-------|
| 0 → 0 | 6 → 0 |
| 1 → 1 | 7 → 1 |
| 2 → 2 | 8 → 2 |
| 3 → 3 | — |
| 4 → 4 | — |
| 5 → 5 | — |

→ 결과 `0, 1, 2` 가 각각 2회, `3, 4, 5` 는 1회. **낮은 구간 확률 2배**. rand 자체는 균등해도 `%` 를 먹이면 **비균등 분포**로 변함.

메이플 케이스:
- rand 약 42억, divisor 10억 → 몫 4, 나머지 ~2.9억.
- 0 ~ 2.9억 구간이 나머지 구간보다 **한 번씩 더** 등장.
- 비율: 대략 `5/4` 배. **드랍률 29% 이하 아이템은 표기 대비 실제가 25% 가량 부풀려짐**.

### 드롭률 버프 고장 현상

- 코드 추정: `if (rand()%10억 < 임계값)` 에서 임계값 상수만 조정하는 방식.
- 예: 20% 아이템 → 임계값 `2억`. +10% 버프 → 임계값 `2.2억` 로 변경.
- **편향 구간(0~2.9억) 에 붙는 과대표집 질량이 상수 변경 증분을 집어삼켜** 결과 확률이 거의 안 움직임.
- 영상 실험: 대략 **17% 이상 상향**해야 유의미한 확률 상승. +5/+10% 버프는 무효.

### 해결책 4가지

1. **품질 좋은 최신 PRNG** 사용 — 균등 분포, 범위 직접 지정. C++ `std::uniform_int_distribution`, xoshiro, PCG 계열.
2. **`rand()` 쓰되 모듈로 피하기** — `rand() == 0` 식으로 특정 정수와 직접 비교 (범위가 맞으면).
3. **rand_max 와 divisor 의 비가 크면 그냥 써도 OK** — 편향 질량이 무시할 수준.
4. **Rejection sampling** — 편향 구간에 떨어지면 버리고 다시 뽑기. 정확.

## 내가 주목한 부분

- **옛 게임 확률 버그의 고전 원인이 수학이 아니라 `%` 문법** 이라는 점. 난수 생성기 자체가 아니라 **매핑 단계에서 균등성이 깨지는** 구조적 함정.
- **유저 개이득 방향으로만 편향이 작동했다**는 현상. 낮은 드랍률일수록 과대표집 → **희귀템일수록 더 잘 떨어짐**. 운영사 손해.
- **버프 임계값 수정 = 확률 증가** 라는 개발자 직관이 편향 환경에서 **깨진다**. QA 가 샘플 사이즈 작으면 버프 미작동을 놓치기 쉬움.
- **`rand()` 생태 종료 권고** — 현대 언어/라이브러리는 모두 균등 정수 분포 API 를 별도 제공. `%` 관행은 레거시.

## 모순 · 주의

> [!question] 메이플 실제 PRNG
> 영상은 "rand 를 썼는지 뭘 썼는지는 알 수 없으니까 편의상 rand 라 가정" 이라 명시. 실제 메이플 코드베이스의 난수 함수는 비공개 — 결론 구조는 동일하나 **구체 rand_max 값은 추정**.

> [!note] 42억 수치
> 영상이 "42억 9천" 으로 표현한 수치는 `2^32 = 4,294,967,296` ≈ 42.9억 으로 **unsigned 32-bit 최대값**. 일반 C `RAND_MAX` 는 구현마다 다름 (POSIX 최소 32767). 메이플 내부 함수는 32-bit 출력을 가정한 것으로 보임.

> [!question] 17% 경계
> "17% 이상 올려야 유의미" 숫자는 영상 저자가 카운팅 실험으로 구한 경험값. 정확 경계는 `10억 mod (2^32)` 크기와 원 임계값 위치에 의존 — 해석적 산출 필요 시 `편향 구간 크기 / 전체 구간` 비율로 재계산.
