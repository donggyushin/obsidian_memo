# Advisor 패턴 — AI 에이전트 아키텍처의 패러다임 전환

## 핵심 딜레마: 성능 vs 비용

| 모델 | 성능 | 비용 |
|------|------|------|
| Opus | 높음 | 비쌈 |
| Sonnet | 충분 | 저렴 |

기존엔 둘 중 하나를 선택해야 했음. Advisor 패턴은 이 트레이드오프를 해결한다.

---

## Advisor란?

> **작은 모델(Executor)이 전체 작업을 실행하다가, 고차원적 판단이 필요한 순간에만 큰 모델(Advisor)에게 조언을 구하는 구조.**

![[스크린샷 2026-04-21 오후 2.47.50.png]]

### 동작 방식
- Executor(Sonnet)가 작업 전체를 수행
- 스스로 해결하기 어려운 판단이 필요한 순간, Advisor(Opus)에게 질문
- Advisor는 **도구를 사용하지 않고**, **사용자에게 출력물도 생성하지 않음** — 오직 Executor에게 조언만 전달
- 단일 API로 통합 — 복잡한 오케스트레이션 로직 불필요

---

## Advisor vs 서브에이전트

| 구분 | 기존 서브에이전트 | Advisor |
|------|-----------------|---------|
| 오케스트레이터 | 큰 모델(Opus)이 작업 분배 | 작은 모델(Sonnet)이 실행 주도 |
| 흐름 | Top-down | Bottom-up |
| 비용 | 전체 작업에 Opus 요금 | 판단 순간(10~20%)에만 Opus 요금 |

**결과:** Sonnet 단독보다 높은 성능, Opus 단독보다 낮은 비용 → **저비용 고효율**

## Claude Code에서 사용하는 법

```
/advisor   →  Advisor 모델을 Opus로 설정, 기본 모델은 Sonnet 4.6 설정
```

### 권장 사용 시점

1. **작업 시작 전** — 방향 설정, 접근법 검토
2. **작업 완료 후** — 결과물 검증, 누락 항목 확인
3. **막혔을 때** — 반복 오류, 방향 전환 고려 시

## 아키텍처 패러다임의 전환

> "정답은 더 큰 모델이 아니라, 더 똑똑한 시스템 아키텍처다." — Anthropic

앞으로의 AI 에이전트 설계 공식:
- **계층형 모델 조합** — 역할에 맞는 모델을 선택적으로 사용
- **판단의 밀도** 에 따라 모델 등급을 동적으로 배분
- 단일 최강 모델 vs 협력하는 모델 시스템 → 후자가 더 경제적이고 실용적
